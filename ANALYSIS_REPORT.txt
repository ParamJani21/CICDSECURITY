================================================================================
CICDSECURITY - COMPREHENSIVE PROJECT ANALYSIS REPORT
================================================================================

EXECUTIVE SUMMARY
================================================================================

CICDSECURITY is a Flask-based security scanning orchestration dashboard that 
automates the execution of multiple security tools (OpenGrep, Trivy, 
TruffleHog) against GitHub repositories. It leverages GitHub App authentication 
to clone private repos and runs selectable scans (SATS, SBOM, SECRET) with 
results aggregation and historical tracking.

Project Type: Flask Web Application (not a library)
Architecture: Modular Python backend with Jinja2 templates and JavaScript frontend
Database: SQLite with SQLAlchemy ORM
Execution Environment: Linux/WSL with external tool integration
Status: Production-ready with complete functionality


================================================================================
PART 1: OVERALL PURPOSE AND ARCHITECTURE
================================================================================

Core Mission:
  1. Authenticate via GitHub App (JWT-based OAuth)
  2. Clone repositories using installation tokens
  3. Run multiple security analysis tools in sequence
  4. Merge findings from all tools into unified report
  5. Store results and provide web dashboard for inspection

System Components:
  - Flask Web Server (port 5000)
  - SQLite Database (cicdsecurity.db)
  - GitHub API Integration (JWT authentication)
  - Security Tool Pipeline (WSL/Linux execution)
  - Result Storage & Retrieval (logs/tool-output/{scan_id}/)
  - User Authentication & Authorization (bcrypt, sessions)

2.4 CORE SCAN ORCHESTRATION
---

File: modules/control_apis.py (1,504 lines)

Responsibilities:
  - Clone repositories using GitHub App authentication
  - Run external security scanning tools
  - Merge findings from all tools into unified format
  - Save results to disk
  - Clean up temporary files

Key Functions:
  clone_repository()    (~200 lines) - Clone repo via WSL with auth token
  run_opengrep_scan()   (~120 lines) - Execute OpenGrep for code analysis
  run_truffle_scan()    (~80 lines)  - Execute TruffleHog for secret scanning
  run_trivy_scan()      (~120 lines) - Execute Trivy for SBOM generation
  merge_findings()      (~180 lines) - Combine findings from all tools
  save_scan_results()   (~120 lines) - Write JSON files
  trigger_scan()        (~250 lines) - Main orchestration (6-step workflow)
  cleanup_cloned_repo() (~80 lines)  - Remove cloned repo from /tmp

WSL Execution:
  - Detects if WSL is available
  - On Windows: runs via 'wsl -e bash -c command'
  - On Linux: runs directly via 'bash -c command'
  - Supports timeouts (default 300 seconds, 600 for scans)
  - Captures stdout/stderr for parsing

6-Step Scan Workflow:
  STEP 1: CLONE REPOSITORY
    - Get GitHub App installation token
    - Construct authenticated HTTPS URL with token
    - Run: git clone --branch {branch} --depth 1 --single-branch {url}
    - Verify clone successful

  STEPS 2-4: RUN SCANS (based on scan_types parameter)
    - If 'sats' in scan_types: Run OpenGrep (semgrep --json --config=auto .)
    - If 'secret' in scan_types: Run TruffleHog (trufflehog filesystem . --json)
    - If 'sbom' in scan_types: Run Trivy (trivy sbom --format cyclonedx .)

  STEP 5: MERGE FINDINGS
    - Normalize severity levels (CRITICAL/MEDIUM/LOW)
    - Deduplicate by (file, line, issue_type)
    - Exclude .git/ artifacts (auth token exposure)
    - Combine sources (track which tools found same issue)
    - Generate summary (counts by severity/category)

  STEP 6: SAVE RESULTS
    - Create logs/tool-output/{scan_id}/ directory
    - Write opengrep.json, truffle.json, trivy.json
    - Write merged.json (unified findings)
    - Verify all files written successfully

  STEP 7: CLEANUP
    - Remove /tmp/{owner}/{repo}/ directory

PART 4: DATA FLOW - SCAN TRIGGER TO RESULT STORAGE
================================================================================

Scan Initiation:
  User clicks "Scan Repository"
  POST /api/repos/scan
  {
      "repo_id": "123",
      "repo_name": "my-repo",
      "repo_owner": "my-org",
      "repo_url": "https://github.com/my-org/my-repo.git",
      "repo_branch": "main",
      "scan_types": ["sats", "sbom", "secret"]
  }

Backend Processing Flow:
  1. Route Handler (app/routes.py)
     - Validate request parameters
     - Call trigger_scan()

  2. Scan Controller (modules/scan_controller.py)
     - Wrapper that imports from control_apis.py

  3. Core Orchestration (modules/control_apis.py::trigger_scan)
     - Step 1: Clone repository
     - Steps 2-4: Run scans (OpenGrep, TruffleHog, Trivy)
     - Step 5: Merge findings
     - Step 6: Save results
     - Step 7: Cleanup

Step 1: Clone Repository
  - Get GitHub App installation token
  - Construct authenticated URL: https://x-access-token:{token}@github.com/...
  - Run: git clone --branch {branch} --depth 1 --single-branch {url}
  - Verify clone, get commit info
  - Output: clone_path, commit_info

Step 2-4: Run Scans (selective based on scan_types)
  OpenGrep (SATS):
    Command: semgrep --json --config=auto . 2>&1 || true
    Output: JSON array of findings
    Timeout: 600 seconds

  TruffleHog (SECRET):
    Command: trufflehog filesystem . --json --no-update 2>&1 || true
    Output: JSON lines with detector results
    Timeout: 600 seconds

  Trivy (SBOM):
    Command: trivy sbom --format cyclonedx . 2>&1 || true
    Output: CycloneDX JSON format
    Timeout: 600 seconds

Step 5: Merge Findings
  - Normalize severity: ERROR/CRITICAL/HIGH -> CRITICAL
  - Deduplicate: same (file, line, type) from multiple tools -> one entry
  - Exclude: .git/ directory artifacts (auth tokens)
  - Combine sources: track which tools found each issue
  - Aggregate: count by severity, category, tool

Step 6: Save Results
  Directory: logs/tool-output/{scan_id}/
  Files:
    - merged.json (unified findings - primary result)
    - opengrep.json (raw OpenGrep output)
    - truffle.json (raw TruffleHog output)
    - trivy.json (raw Trivy SBOM output)

Step 7: Cleanup
  - Remove /tmp/{owner}/{repo}/ directory recursively
  - Clean empty owner directory if no other repos

Result Structure (merged.json):

PART 5: GITHUB APP AUTHENTICATION
================================================================================

Authentication Flow:
  1. LOAD CREDENTIALS
     - Read from database (encrypted) or .env
     - GITHUB_APP_ID
     - GITHUB_SECRET_KEY (RSA private key)

  2. GENERATE JWT
     - Create JWT payload:
       {
           "iat": {now},
           "exp": {now + 300 seconds},  // 5 minutes (GitHub requirement)
           "iss": {GITHUB_APP_ID}
       }
     - Sign with RSA private key (RS256 algorithm)

  3. REQUEST INSTALLATIONS
     - GET /app/installations
     - Headers: Authorization: Bearer {JWT}
     - Response: [installation_id_1, installation_id_2, ...]

  4. GET INSTALLATION TOKEN
     - POST /installations/{installation_id}/access_tokens
     - Headers: Authorization: Bearer {JWT}
     - Response: {token: "ghu_...", expires_at: "..."}

  5. CLONE REPOSITORY
     - git clone https://x-access-token:{token}@github.com/owner/repo.git
     - Token grants read access to private repos

Key Security Considerations:
  - JWT expires after 5 minutes (minimizes token exposure)
  - Private key stored encrypted in database by default
  - Can fall back to .env for single-user deployments
  - Escaped newlines (\n) handled by env_config.py
  - Installation tokens have limited scope (repo access only)
  - Uses GitHub-recommended x-access-token cloning scheme

PART 6: THE THREE SCAN TYPES AND THEIR TOOLS
================================================================================

6.1 SATS - STATIC APPLICATION TESTING (OPENGREP + SLITHER)
---

Scan Type ID: sats

Purpose: Detect security issues in source code without execution

Primary Tool: OpenGrep (formerly Semgrep)
  - Pattern-based static analysis
  - Configurable rule sets
  - Supports 30+ programming languages

Integration:
  - Detect available tool (prefer opengrep over semgrep)
  - Run with auto config: semgrep --json --config=auto .
  - Parse JSON results and extract findings

Tool Installation:
  apt-get install opengrep
  OR: go install github.com/PatrickKhanz/owasp-gpt@latest
  OR: pip install semgrep

Detection Categories:
  - SQL Injection
  - XSS (Cross-Site Scripting)
  - Hardcoded Secrets
  - Insecure Cryptography
  - Code Injection
  - Path Traversal
  - LDAP Injection

Risk Levels:
  - ERROR -> CRITICAL
  - WARNING -> MEDIUM
  - INFO -> LOW


6.2 SBOM - SOFTWARE BILL OF MATERIALS (TRIVY)
---

Scan Type ID: sbom

Purpose: Generate inventory of dependencies and components

Tool: Trivy by AquaSecurity
  - Fast, reliable SBOM generation
  - Supports multiple formats (SBOM CycloneDX, JSON)
  - Integrates with supply chain security

Installation:
  wget https://github.com/aquasecurity/trivy/releases/download/v0.51.0/trivy_0.51.0_Linux-64bit.tar.gz
  tar zxvf trivy.tar.gz -C /usr/local/bin/

Integration:
  - Run SBOM scan only (not vulnerability scan)
  - Command: trivy sbom --format cyclonedx . 2>&1 || true
  - Parse CycloneDX JSON format

Output Data:
  - Component name, version, type
  - Package URL (purl) for supply chain tools
  - License information
  - Dependency relationships


6.3 SECRET - SECRET SCANNING (TRUFFLEHOG)
---

Scan Type ID: secret

Purpose: Detect exposed secrets, API keys, credentials

Tool: TruffleHog v3 by Truffle Security
  - Pattern matching for 100+ detector types
  - GitHub token validation
  - AWS key verification
  - Database credentials detection

Installation:
  go install github.com/trufflesecurity/trufflehog/v3@latest
  OR: pip install trufflehog

Integration:
  - Run filesystem scan
  - Command: trufflehog filesystem . --json --no-update 2>&1 || true
  - Parse JSON lines (each line is a detector result)

Detector Types:
  - AWS Keys (Access Key, Secret Key)
  - GitHub Tokens (Classic, PAT, OAuth)
  - Private Keys (RSA, DSA, EC)
  - Database Credentials (MySQL, PostgreSQL, MongoDB)
  - API Keys (Slack, Discord, SendGrid, etc.)
  - Mailchimp API Key
  - Stripe API Key
  - Twilio API Key

Risk Levels:
  - Private Key, GitHub Token -> CRITICAL
  - AWS Key, Database Credential -> HIGH
  - Other secrets -> MEDIUM to HIGH

PART 8: KEY QUIRKS AND IMPORTANT NOTES
================================================================================

WSL Path Conversion:
  Windows paths: C:\Users\user\CICDSECURITY
  WSL paths: /mnt/c/Users/user/CICDSECURITY
  Auto-converted by get_wsl_path() function
  Always quote paths with spaces in shell commands

Token Management:
  JWT tokens expire after 5 minutes (GitHub requirement)
  Always call get_github_app_token() before API calls
  Installation tokens are short-lived, requested per-clone

RSA Private Key Format:
  Must have actual newlines (not escaped \n)
  env_config.py automatically converts \n <-> actual newlines
  Must be in PEM format: -----BEGIN RSA PRIVATE KEY-----

Scan Failure Behavior:
  OpenGrep failure: Stops workflow, returns error (critical tool)
  TruffleHog failure: Continues workflow, returns skipped (non-critical)
  Trivy failure: Continues workflow, returns skipped (non-critical)

.git/ Directory Filtering:
  Findings from .git/ are excluded from merged results
  Reason: Auth tokens stored in .git/config during clone
  These are artifacts of scanning, not repo findings

Tool Output Parsing:
  OpenGrep: Handles both full JSON parse and line-wise JSON
  TruffleHog: Expects JSON lines (one JSON object per line)
  Trivy: Skip log lines, parse JSON starting from {

Cloning Behavior:
  Uses shallow clone: --depth 1 --single-branch
  Reduces clone time and disk space
  Only fetches specified branch

Results Storage Workflow:
  Individual tool results: {scan_id}/{tool}.json
  Merged findings: {scan_id}/merged.json
  All files written before cleanup
  If write fails, cleanup still occurs, returns error

CICDSECURITY is a comprehensive, production-ready security scanning 
orchestration platform that:

1. Automates Security Scanning across multiple tools and formats
2. Integrates with GitHub via App authentication for seamless repo access
3. Provides Web Dashboard for easy result viewing and filtering
4. Stores Historical Data for tracking security improvements over time

6. Supports Multiple Scan Types (SATS, SBOM, SECRET) with optional selection

tool additions.
